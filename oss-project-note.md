# OSS プロジェクトのメモ

## フォルダ名

「tests/? or test/?」みたいなやつ。
どうも日本語話者にはよくわからないノリがあるらしい。

GPT-5 にまとめてもらった。正しいとは限らないけど、参考まで。

| フォルダ名   | 命名の一般ルール / 意味合い                                    | 主な利用エコシステム / ツール               | OSS での採用傾向(定性的)   | 備考                                                                  |
| ------------ | -------------------------------------------------------------- | ------------------------------------------- | -------------------------- | --------------------------------------------------------------------- |
| `src/`       | 「ソースコードの集合」を表す**カテゴリ名**なので単数形が一般的 | Node.js/TS、Python(一部)、Rust、C/C++、Java | **非常に一般的**           | Java は `src/main/java` を標準として細分化(Maven/Gradle)              |
| `dist/`      | 「配布物(distribution)」を表すカテゴリ名で単数形               | Web/JS バンドラ(Vite、Rollup、tsup など)    | **非常に一般的**           | `build/` を使うプロジェクトも多い。`out/` や `target/` は言語依存     |
| `build/`     | ビルド成果物/中間成果物の出力                                  | C/C++、Java(Gradle)、Go(一部)               | **一般的**                 | Java では `build/`(Gradle) vs `target/`(Maven)                        |
| `public/`    | 静的公開資産(直配信されるもの)                                 | React、Vite、Next.js(`public/`)             | **一般的**                 | Web 系でよく使われる。`static/` との使い分けはフレームワーク次第      |
| `assets/`    | 画像/フォント/音声など複数種の**静的リソース**(複数形)         | Web 全般、ゲーム、フロントエンド            | **一般的**                 | 下位に `images/`, `fonts/`, `icons/` 等の**複数形**が並ぶ傾向         |
| `images/`    | 画像群(複数ファイルの集合なので複数形)                         | Web/ドキュメント                            | **一般的**                 | `img/` より `images/` が可読性高く推奨されることが多い                |
| `docs/`      | ドキュメント(複数形)                                           | OSS 全般、Sphinx、MkDocs、Docusaurus        | **非常に一般的**           | `doc/` より `docs/` が主流。サイト生成ツールも `docs/` を前提にしがち |
| `tests/`     | テスト**一式**(複数ファイル前提で複数形)                       | Python(pytest 推奨)、Node.js、Rust、C/C++   | **非常に一般的**           | Python 公式の「Good Practices」でも `tests/` を推奨                   |
| `test/`      | テストの置き場(単数形)                                         | 小規模プロジェクトや個人開発で散見          | **やや少数派**             | 習慣としては `tests/` が優勢                                          |
| `__tests__/` | 「テスト専用ディレクトリ」を明示する慣習(Jest 等)              | Jest、Vitest(互換)、React 周辺              | **一般的(Jest 文化圏)**    | ルート置き/`src/__tests__/` の両スタイルあり                          |
| `examples/`  | 使用例やサンプル(複数形)                                       | ライブラリ/CLI/SDK                          | **一般的**                 | `example/` より `examples/` が通例                                    |
| `scripts/`   | 補助スクリプト群(複数形)                                       | OSS 全般                                    | **一般的**                 | CI 補助、リリース補助など。`tools/` と分担することも                  |
| `bin/`       | 実行可能ファイル/エントリポイント                              | Node.js(npm の `bin`)、Unix 系              | **一般的**                 | Node パッケージでは `package.json` の `bin` と連動                    |
| `lib/`       | ライブラリコード(歴史的慣習)                                   | Node.js(旧来)、Ruby、Perl                   | **減少傾向/レガシー寄り**  | 近年は `src/` が主流、`lib/` は互換保持で残ることあり                 |
| `packages/`  | モノレポ内の各パッケージ(複数形)                               | pnpm/Workspaces、Yarn、Lerna、Rush          | **非常に一般的(モノレポ)** | `apps/` と併用(アプリとライブラリで階層分離)                          |
| `tools/`     | ツール/スクリプト群(複数形)                                    | OSS 全般                                    | **一般的**                 | `scripts/` より「ツール置き場」感が強い。用途で使い分け               |
